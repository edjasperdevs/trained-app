  <action>
In the `validateCode` function (lines 40-112), remove the dev fallback block that accepts any 8+ character code when Supabase is not configured (lines 53-64).

The current code has:
```typescript
if (!supabase) {
  if (import.meta.env.DEV) {
    // DEV only: accept any 8+ char code when Supabase not configured
    console.log('[Access] Supabase not configured, using dev fallback')
    set({ hasAccess: true, ... })
    return { success: true }
  }
  return { success: false, error: 'Validation unavailable. Please try again later.' }
}
```

Replace this entire block with:
```typescript
if (!supabase) {
  return {
    success: false,
    error: 'App configuration error. Please contact support.'
  }
}
```

This ensures that missing Supabase configuration is treated as a configuration error (not a bypass), both in development and production. Developers can still test locally by providing proper VITE_SUPABASE_URL and VITE_SUPABASE_ANON_KEY environment variables.

Update the comment on line 48-49 to reflect that the master code check is server-side only (already correct), and add a comment above the new block explaining why dev fallback is not provided:

```typescript
// No dev fallback: misconfigured environment should fail explicitly
// rather than allowing unauthorized access. Developers should configure
// VITE_SUPABASE_URL and VITE_SUPABASE_ANON_KEY for local testing.
if (!supabase) {
  return {
    success: false,
    error: 'App configuration error. Please contact support.'
  }
}
```
