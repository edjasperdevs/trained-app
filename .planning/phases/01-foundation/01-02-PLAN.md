**Step 2: Replace each hardcoded color with the appropriate token reference**

The research identified 67 hardcoded colors across 16 files. Here is the mapping strategy:

| Hardcoded Value | Token Replacement | Notes |
|---|---|---|
| `text-white` | `text-text-on-primary` or `text-text-primary` | Use `text-on-primary` when on colored backgrounds (buttons), `text-primary` when on dark backgrounds |
| `bg-black` | `bg-background` | App background |
| `text-gray-400` / `text-gray-500` | `text-text-secondary` | Secondary text |
| `text-gray-300` | `text-text-primary` | Primary text on dark |
| `bg-gray-800` / `bg-gray-900` | `bg-surface` or `bg-surface-elevated` | Surface colors |
| `border-gray-700` / `border-gray-600` | `border-border` | Border color |
| Inline `#hex` values | Corresponding `var(--color-*)` or Tailwind token class | Match to nearest token |
| Inline `rgba()` for chart colors | CSS custom properties: `var(--color-primary)` etc. | For WeightChart.tsx specifically |
| `bg-[#0a0a0f]` (main.tsx ErrorFallback) | `bg-background` | |
| `bg-[#00f5d4]` (main.tsx ErrorFallback) | `bg-primary` | This was a GYG color -- replace with primary |
| `text-black` (main.tsx ErrorFallback) | `text-text-on-primary` | Text on primary button |

**Known high-density files from research:**
- `WeightChart.tsx` -- 10 inline rgba() for chart line/area colors. Replace with CSS variables. For chart-specific colors that don't have exact tokens, create chart-specific CSS custom properties in a `@layer components` block in index.css OR use the closest token with opacity modifier (e.g., `var(--color-primary)` with Tailwind opacity).
- `Onboarding.tsx` -- 7 hardcoded hex values (these may be in gradient backgrounds or accent colors)
- `Auth.tsx` -- 5 hardcoded values
- `BadgeUnlockModal.tsx` -- 9 hardcoded values (particle/glow effect colors)
- `src/main.tsx` -- ErrorFallback has `bg-[#0a0a0f]`, `text-white`, `text-gray-400`, `bg-[#00f5d4]`, `text-black`

**Step 3: Handle special cases**

For `WeightChart.tsx` chart colors that need specific rgba values:
- If the chart library (likely inline SVG/canvas) needs direct color strings, use `getComputedStyle(document.documentElement).getPropertyValue('--color-primary')` pattern, OR define chart-specific CSS custom properties in index.css:
  ```css
  /* Chart-specific token overrides */
  :root {
    --chart-line: var(--color-primary);
    --chart-area: rgba(213, 85, 80, 0.1);
    --chart-grid: var(--color-border);
  }
  ```
  Then reference these in WeightChart.tsx.

For `BadgeUnlockModal.tsx` particle effects:
- Glow/particle colors should reference `var(--color-primary)`, `var(--color-warning)` etc.
- If values are passed as JavaScript strings to animation configs, use CSS custom property reads or define constants that reference the token values.

**Step 1: Install Fontsource variable font packages**
```bash
npm install @fontsource-variable/inter @fontsource-variable/oswald @fontsource-variable/jetbrains-mono
```

**Step 2: Add font imports to src/main.tsx**
Add these imports BEFORE the `./index.css` import (so fonts are available when CSS loads):
```typescript
import '@fontsource-variable/inter'
import '@fontsource-variable/oswald'
import '@fontsource-variable/jetbrains-mono'
```

**Step 3: Update @theme font family names in src/index.css**
Fontsource variable font packages register with "Variable" suffix. Update the @theme font tokens:
```css
--font-heading: 'Oswald Variable', 'Oswald', sans-serif;
--font-body: 'Inter Variable', 'Inter', sans-serif;
--font-mono: 'JetBrains Mono Variable', 'JetBrains Mono', monospace;
```
Keep the non-Variable names as fallbacks in case the variable fonts fail to load.

**Step 4: Remove Google Fonts from index.html**
Delete these three lines from `<head>`:
```html
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link href="https://fonts.googleapis.com/css2?family=Oswald:wght@400;500;600;700&family=Inter:wght@400;500;600;700;800&family=JetBrains+Mono:wght@400;500;600;700&display=swap" rel="stylesheet">
```

**Step 5: Verify Google Fonts caching was removed from vite.config.ts**
Plan 01 should have already removed the workbox runtimeCaching entries for `fonts.googleapis.com` and `fonts.gstatic.com`. If they are still present, remove them now. These entries would cause the service worker to try caching requests that no longer happen.

**Step 6: Upgrade framer-motion to motion**
```bash
npm uninstall framer-motion
npm install motion
```

**Step 7: Find and replace all framer-motion imports**
Search for all files importing from framer-motion:
```bash
grep -rn "from 'framer-motion'" src/
grep -rn "from \"framer-motion\"" src/
```

Replace every occurrence:
- `from 'framer-motion'` becomes `from 'motion/react'`
- `from "framer-motion"` becomes `from "motion/react"`

This is a safe find-and-replace -- the motion/react package exports the exact same API (motion, AnimatePresence, useAnimation, useReducedMotion, etc.).

**Step 8: Update vite.config.ts manualChunks**
Change the vendor chunk:
```typescript
'vendor-motion': ['motion']  // was: ['framer-motion']
```

**Step 9: Verify animations still work**
```bash
npm run build
npm run dev
```

Check that:
- Button press animations work (whileTap scale)
- Card hover animations work
- Page transitions animate (AnimatePresence)
- Navigation indicator animates (layoutId)
- Any shimmer/pulse animations play

**What to avoid and WHY:**
- Do NOT change any animation VALUES (durations, spring configs, scale amounts) -- that is Phase 5 work. Only change the import path.
- Do NOT remove any animation code -- only update imports from framer-motion to motion/react.
- Do NOT change the Fontsource font family CSS fallbacks to use ONLY the Variable name -- keep the non-Variable name as a fallback for robustness.
- Do NOT remove the `font-display: swap` behavior -- Fontsource handles this internally, but verify fonts appear quickly.
  </action>
