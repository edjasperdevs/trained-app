Key architectural decisions from research:
- Use VITE_DEV_BYPASS=true to skip auth gates for most tests (auth flow tested separately in Phase 2)
- Seed Zustand stores via localStorage (not Supabase) because app is offline-first
- Block service workers in Playwright config (serviceWorkers: 'block')
- Disable animations via CSS override in test setup
- E2E tests live in /e2e/ directory (separate from Vitest unit tests in /src/)
- Zustand persist format: { state: {...}, version: N } with keys like 'gamify-gains-user'
- devSeed.ts already has comprehensive seed data for all 8 stores

Auth wall order: AccessGate (access code) -> Auth (email/password) -> Onboarding (if new user)
Dev bypass: `import.meta.env.VITE_DEV_BYPASS === 'true'` skips all three gates

Screens requiring data-testid attributes:
- Navigation: 5 nav links (home, workouts, macros, avatar, settings)
- AccessGate: access code input, submit button
- Auth: email input, password input, sign in button, sign up button, toggle mode link
- Onboarding: step-specific inputs, next/back buttons, progress indicator
- Home: check-in button, streak display, XP display, level display
- Workouts: start workout button, exercise cards, set inputs, complete button
- Macros: food search input, meal log entries, macro totals display, add meal button
- AvatarScreen: avatar display, evolution stage, mood indicator
- Settings: various settings controls, export button, sign out button
- CheckInModal: confirm button, streak display
- XPClaimModal: claim button, XP amount display

     /playwright-report/
     /blob-report/

     - `seedStore(page, key, state, version?)` function that uses `page.addInitScript` to set a single store in localStorage with the Zustand persist envelope format `{ state, version: version ?? 0 }`
     - `seedAllStores(page)` function that seeds ALL 8 stores with realistic test data. Adapt the seed data directly from `src/lib/devSeed.ts` -- copy the store state objects (userStore, workoutStore, macroStore, xpStore, avatarStore, achievementsStore, accessStore, remindersStore). Use static values where devSeed uses dynamic ones (e.g., hardcode today's date math results). The critical fields for bypassing auth gates:
       - `access` store: `{ hasAccess: true, licenseKey: 'TEST-E2E-KEY' }`
       - `user` store: `{ profile: { onboardingComplete: true, username: 'E2ETestUser', ... } }`
     - `clearAllStores(page)` function that removes all `gamify-gains-*` keys from localStorage

  2. Create `e2e/fixtures/base.ts` with:
     - Extended Playwright test using `test.extend<{ seededPage: Page }>`
     - `seededPage` fixture that:
       a. Calls `seedAllStores(page)` to inject all Zustand stores
       b. Adds a CSS style tag to disable animations: `*, *::before, *::after { animation-duration: 0s !important; transition-duration: 0s !important; }` (Pitfall #7)
       c. Navigates to `'/'`
       d. Waits for the page to be ready (the app renders its main content)
       e. Yields the page for the test
     - Export `test` (extended) and `expect` from Playwright

  3. Create `e2e/tests/smoke.spec.ts` with:

       - Assert the navigation bar is visible (look for `nav` element with aria-label "Main navigation")
     - Test: "can navigate to all main screens"
       - Use `seededPage` fixture
       - Click each nav link (Home, Workouts, Macros, Avatar, Settings) using `data-testid` attributes (added in Task 3)
       - Assert each screen renders its primary content (use `data-testid` attributes for key elements)
     - Test: "test isolation -- fresh context has no data"
       - Use a regular (non-seeded) page
       - Navigate to baseURL
       - Assert the app shows the AccessGate or Auth screen (since no data is seeded and VITE_DEV_BYPASS is true, but access store is empty). Actually, with VITE_DEV_BYPASS=true the app skips all gates. So instead: assert that navigating to '/' renders the home screen layout (even if empty state) -- this proves VITE_DEV_BYPASS works and each test has its own context.

  IMPORTANT: `seedAllStores` must use `page.addInitScript` (NOT `page.evaluate`) so the data is in localStorage BEFORE the page loads and Zustand hydrates from it. The `addInitScript` runs before any page script executes.
