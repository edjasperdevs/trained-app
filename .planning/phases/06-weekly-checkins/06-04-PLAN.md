Add a check-in response pull section to the existing `pullCoachData()` function in sync.ts.

After the existing assigned workouts pull block (around line 516), add:

```typescript
// --- Pull latest check-in response ---
const { data: latestCheckin } = await client
  .from('weekly_checkins')
  .select('id, week_of, status, coach_response, reviewed_at')
  .eq('client_id', user.id)
  .order('week_of', { ascending: false })
  .limit(1)
  .single()

if (latestCheckin) {
  // Store the latest check-in status in localStorage for the Home screen to read
  // Using a simple localStorage key (not Zustand) to avoid adding a new store for one piece of data
  localStorage.setItem('trained-latest-checkin', JSON.stringify({
    id: latestCheckin.id,
    week_of: latestCheckin.week_of,
    status: latestCheckin.status,
    coach_response: latestCheckin.coach_response,
    reviewed_at: latestCheckin.reviewed_at,
  }))
}
```

**Why localStorage instead of a Zustand store:** This is a single read-only value that doesn't need reactive state management. The Home screen reads it once on mount. A Zustand store would be overkill for this (following the project pattern of keeping non-persisted sync data lightweight). The key 'trained-latest-checkin' follows the project naming pattern but uses 'trained-' prefix to distinguish from 'gamify-gains-' store keys.

**Alternative approach (preferred if cleaner):** If the useWeeklyCheckins hook already has a way to cache the latest check-in locally (from Plan 01's fetchMyCheckins), then pullCoachData can simply call a function from the hook to refresh the cache. Check the Plan 01 SUMMARY for how fetchMyCheckins stores data.

The key insight: pullCoachData runs on app open, online reconnect, and visibility change. The client's Home screen reads from whatever pullCoachData populated. The flow is:
1. pullCoachData() runs on app open
2. Fetches latest check-in status (submitted or reviewed)
3. Stores in localStorage
4. Home screen reads from localStorage and shows appropriate banner
