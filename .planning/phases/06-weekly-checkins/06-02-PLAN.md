**Auto-populated data computation (CHECK-02):**
At submission time (not render time -- it's a snapshot), compute:
- `auto_weight_current`: Latest entry from userStore.weightHistory
- `auto_weight_weekly_avg`: Average of last 7 days from weightHistory
- `auto_weight_change`: Current week avg minus prior week avg (null if insufficient data)
- `auto_step_avg`: null (app doesn't track steps -- see research pitfall #2)
- `auto_macro_hit_rate`: Percentage of last 7 days where protein >= target AND calories are within 10% of target. Use macroStore.dailyLogs and macroStore.targets.
- `auto_cardio_sessions`: null (app doesn't distinguish cardio -- see research)
- `auto_workouts_completed`: Count of completed workout logs in last 7 days from workoutStore.workoutLogs

Write a `computeAutoData()` function inside the file that reads from stores using getState() (not hooks -- this runs on submit, not render).

**Submission flow:**
1. Validate: at least one field must have content (not all fields required per research anti-patterns)
2. Compute auto data via computeAutoData()
3. Call submitCheckin(formData, autoData) from useWeeklyCheckins hook
4. On success: show toast.success('Check-in submitted!'), navigate back to '/'
5. On error: show toast.error with friendlyError(), stay on page

**State management:** Simple useState for each field. No form library needed (research says useState with section objects is fine for 16 fields). Group by section objects for cleanliness:
```
const [nutrition, setNutrition] = useState({ water_intake: '', caffeine_intake: '', ... })
```

**Styling:** Follow existing design system. Card for each section. Use existing components: Input, Textarea, Label from shadcn/ui. Use the premium dark theme (bg-card, text-foreground, etc.).

**Important:** Do NOT import from CheckInModal.tsx -- this is a completely separate feature (research pitfall #1). Use different icon (ClipboardCheck not Sparkles) and different accent color (secondary or info, not primary) to differentiate.
