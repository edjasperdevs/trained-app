## CRITICAL Issues (16)

### Security

**SEC-01: Secrets committed to git**
- **Files:** `.env`, `.env.local`, `.env.vercel`, `.env.production.local`
- Supabase anon key, USDA API key, master access code (`EARLYADOPTER`), Vercel OIDC token, and Sentry DSN are all in version-controlled files.
- **Fix:** `git rm --cached` all env files, rotate every exposed secret, purge git history with BFG.

**SEC-02: Master access code embedded in client JS bundle**
- **File:** `src/stores/accessStore.ts:92`
- `VITE_MASTER_ACCESS_CODE` is baked into the Vite build output. Anyone can find it in devtools.
- **Fix:** Move access code validation to a Supabase Edge Function (server-side).

**SEC-03: No frontend route guard on `/coach`**
- **File:** `src/App.tsx:179`
- Any authenticated user can navigate to `/coach`. No `CoachGuard` component exists. The route is explicitly exempt from access gate and onboarding checks.
- **Fix:** Add a `CoachGuard` wrapper that checks `isCoach()` before rendering.

### Code Quality

**CQ-01: React Rules of Hooks violation in Macros.tsx**
- **File:** `src/screens/Macros.tsx:235`
- `useState(false)` for `quickLogSuccess` is declared *after* a conditional early return at line 224. If `targets` toggles from null to defined, React will crash with a hooks ordering error.
- **Fix:** Move all `useState` calls above the conditional return.

### UX / Edge Cases

**UX-01: No double-tap protection on XP-awarding actions**
- **Files:** `src/screens/Workouts.tsx:116-147` (complete workout), `src/screens/Workouts.tsx:91-113` (minimal workout), `src/screens/CheckInModal.tsx:87-152` (daily check-in)
- No `isSubmitting` guard. Double-tapping awards XP twice and may corrupt workout logs.
- **Fix:** Add `isSubmitting` state, disable buttons during async operations.

*(UX-01 covers both the Workouts double-tap and CheckInModal double-tap since they share the same pattern)*

### Performance

**PERF-01: Unbounded localStorage growth**
- **Files:** `src/stores/macroStore.ts` (dailyLogs), `src/stores/workoutStore.ts` (workoutLogs), `src/stores/xpStore.ts` (dailyLogs)
- Arrays grow forever with no pruning. After a year: ~365 macro logs, ~260 workout logs (each with nested exercise/set arrays). Will hit the 5-10MB localStorage limit and cause slow hydration.
- **Fix:** Implement a 90-day rolling window. Archive older data to cloud only.

**PERF-02: Broad store subscriptions cause excessive re-renders**
- **Files:** `src/screens/Workouts.tsx:16-40` (~20 destructured props), `src/screens/Home.tsx:26-32` (4 stores including bare `useAvatarStore()`), `src/screens/Macros.tsx:24-42`
- Every property change in any subscribed store re-renders the entire screen.
- **Fix:** Use individual selectors: `useWorkoutStore(s => s.currentPlan)`. Access actions via `getState()`.

**PERF-03: 30+ sequential HTTP requests on sync**
- **File:** `src/lib/sync.ts:456-460`
- `pushClientData` loops 10 workouts sequentially (each = 2 requests). Combined with 5-6 other sequential calls = 25-30 requests in series. Each `getUser()` is called redundantly per function.
- **Fix:** Parallelize with `Promise.all()`, batch upserts, get user once.

### Accessibility

**A11Y-01: Multiple modals lack `role="dialog"` and `aria-modal`**
- **Files:** `src/screens/Workouts.tsx` (4 modals at lines 583, 630, 1205, 1285), `src/screens/Home.tsx:561`
- Plain `<div>` overlays — screen readers can't identify them as dialogs.
- **Fix:** Add `role="dialog"`, `aria-modal="true"`, `aria-label` to all modal containers.

**A11Y-02: No focus trapping in any modal**
- **Files:** All modal implementations across the codebase
- Zero focus trap implementations found. Keyboard users Tab out of modals into hidden background content.
- **Fix:** Install `focus-trap-react` and wrap all modal content.

**A11Y-03: No `prefers-reduced-motion` support**
- **File:** `src/index.css:115-133,251-253`
- 4 CSS animations + global transitions on all interactive elements. Zero `prefers-reduced-motion` media queries anywhere.
- **Fix:** Add `@media (prefers-reduced-motion: reduce) { *, *::before, *::after { animation-duration: 0.01ms !important; transition-duration: 0.01ms !important; } }`

### Test Coverage

**TEST-01: Zero screen-level component tests**
- 13 screens, 0 unit/component tests. All UI logic (conditional rendering, validation, modal state) is untested at the component level.

**TEST-02: 9 of 12 stores have zero tests**
- Only `macroStore`, `xpStore`, `workoutStore` are tested. `userStore`, `authStore`, `avatarStore`, `achievementsStore`, `remindersStore`, `accessStore`, `syncStore`, `toastStore` have zero coverage.

**TEST-03: Sync push/pull paths untested**
- Only `loadProfileFromCloud` (6 tests). `pushClientData` and `pullCoachData` (the most critical data integrity functions) have zero tests.

## HIGH Issues (30)

### Security

**SEC-04: Incomplete logout — localStorage not cleared**
- **File:** `src/stores/authStore.ts:140-146`
- `signOut()` clears auth state but leaves all Zustand-persisted stores (macros, workouts, XP, profile, weight) in localStorage. On a shared device, the next user sees previous user's health data.
- **Fix:** Clear all persisted store keys on sign-out, or namespace by user ID.

**SEC-05: Data import without validation**
- **File:** `src/screens/Settings.tsx:198-236`
- Import only checks if `version`, `user`, or `xp` exists (any one passes). `src/lib/validation.ts` has proper validators but they're never called during import. Malicious JSON can corrupt stores.
- **Fix:** Apply `isValidMacroTargets`, `isValidWorkoutLog`, etc. from `validation.ts` to each import section.

**SEC-06: PostgREST filter injection via unsanitized search**
- **File:** `src/hooks/useClientRoster.ts:54`
- Search input interpolated directly into `.or()` filter string. Characters like `,`, `.`, `(` can inject filter clauses.
- **Fix:** Escape PostgREST special characters before interpolation.

**SEC-07: `?reset=true` URL destroys all data without confirmation**
- **File:** `src/App.tsx:37-48`
- Any link with `?reset=true` immediately wipes all local data and signs out. No auth check, no confirmation dialog.
- **Fix:** Remove the URL parameter entirely, or add `window.confirm()` + require authentication.

### Code Quality

**CQ-02: Coach.tsx is ~1700 lines with 30+ `useState` calls**
- **File:** `src/screens/Coach.tsx`
- 8+ responsibilities: client roster, client detail modal (5 tabs), invite flow, macro editor, templates, check-in review, workout assignment, intake view.
- **Fix:** Extract `ClientDetailModal`, `ClientListView`, `CheckinReviewView`, `TemplateManagementView`, `InviteClientModal` into separate files.

**CQ-03: `as any` cast on entire Supabase client in intakeApi**
- **File:** `src/lib/intakeApi.ts:11`
- `const db = () => getSupabaseClient() as any` — all database queries bypass TypeScript checking.
- **Fix:** Add intake tables to `database.types.ts`.

**CQ-04: Stale `activeWorkout` local state pattern**
- **File:** `src/screens/Workouts.tsx:46,196-235`
- Active workout copied into `useState`, then manually re-synced after every mutation via `setActiveWorkout(getCurrentWorkout())`. Forgetting one sync = stale UI.
- **Fix:** Derive directly from store: `const activeWorkout = useWorkoutStore(s => s.getCurrentWorkout())`.

**CQ-05: UTC vs local date bug across stores**
- **Files:** `src/stores/macroStore.ts`, `src/stores/workoutStore.ts`, `src/stores/remindersStore.ts`
- Pattern `new Date().toISOString().split('T')[0]` used 25+ times. Returns UTC date — a user logging at 11 PM EST gets the wrong date. Only `userStore` and `xpStore` use the correct `getLocalDateString()`.
- **Fix:** Replace all instances with `getLocalDateString()` from `dateUtils.ts`.

### UX / Edge Cases

**UX-02: Home.tsx coach async calls have no error handling**
- **File:** `src/screens/Home.tsx:50-58`
- `isCoachingClient()` and `hasCheckinForCurrentWeek()` awaited without `.catch()`. If offline or token expired, state silently stays at defaults — weekly check-in banner never shows.
- **Fix:** Add error handling with graceful fallback.

**UX-03: WeeklyCheckIn form data lost on navigation**
- **File:** `src/screens/WeeklyCheckIn.tsx:218-641`
- Many fields across 5 sections. No state persistence to localStorage, no "unsaved changes" warning. Accidental back button = all data lost.
- **Fix:** Persist form state to localStorage or add `beforeunload` warning.

**UX-04: Username saved on every keystroke without validation**
- **File:** `src/screens/Settings.tsx:283-289`
- `setProfile({ username: e.target.value })` on every keystroke. No debounce, no validation — empty strings, whitespace-only, and special characters all accepted.
- **Fix:** Debounce, trim, enforce minimum length.

**UX-05: Quick Log allows 0 and negative values**
- **File:** `src/screens/Macros.tsx:237-247`
- Guard checks if values are truthy strings, but `"0"` is truthy. Negative numbers pass through number inputs.
- **Fix:** Validate `Number(value) > 0` before logging.

**UX-06: Meal deletion has no confirmation dialog**
- **File:** `src/screens/Macros.tsx:484-489, 974`
- Both daily meal and saved meal deletions are immediate and irreversible. No undo capability.
- **Fix:** Add `window.confirm()` or toast-based undo.

**UX-07: Check-in quest always shows incomplete on Home**
- **File:** `src/screens/Home.tsx:161`
- Hardcoded `completed: false` instead of `completed: hasCheckedInToday`.
- **Fix:** Use the actual check-in state.

**UX-08: Coach screen shows bottom nav bar**
- **File:** `src/App.tsx:187`
- Navigation bar renders on all authenticated routes including `/coach`. The coach dashboard is standalone — "Home/Workouts/Macros/Avatar/Settings" nav is confusing.
- **Fix:** Hide nav on `/coach` route.

**UX-09: No re-submission guard on WeeklyCheckIn route**
- **File:** `src/screens/WeeklyCheckIn.tsx:296-345`
- Direct navigation to `/checkin` allows submitting duplicate check-ins. Home screen guards the banner, but the route itself doesn't check.
- **Fix:** Check for existing check-in on mount.

### Performance

**PERF-04: Double-render pattern in Workouts.tsx**
- **File:** `src/screens/Workouts.tsx:212-235`
- Every handler calls a store action (triggers Zustand re-render) then `setActiveWorkout(getCurrentWorkout())` (triggers local state re-render). 2 renders per interaction.
- **Fix:** Derive `activeWorkout` from store selector, eliminating local state.

**PERF-05: `getLastWorkout()` called per exercise per render**
- **File:** `src/screens/Workouts.tsx:883-943`
- Inside `.map()`, each exercise calls `getLastWorkout()` which sorts the full workout history. 5-6 full array sorts per render.

**PERF-06: Sequential sync with redundant `getUser()` calls**
- **File:** `src/lib/sync.ts` (lines 93, 135, 192, 218, 252, 280, 326, 354, 395, 477)
- 8-10 separate `getUser()` calls per sync. Weight logs, saved meals also upserted individually in loops.

**PERF-07: `pullCoachData()` waterfall requests**
- **File:** `src/lib/sync.ts:471-553`
- 4 sequential Supabase queries (getUser, macros, workouts, check-ins) that are independent after auth.
- **Fix:** `Promise.all()` the 3 data queries after getting user.

## Positive Findings

These areas are done well:

- **RLS policies** are solid — `auth.uid()` checks, coach role verification, `prevent_role_change()` trigger
- **No XSS vectors** — no `dangerouslySetInnerHTML` anywhere, all React auto-escaping
- **Navigation component** — proper `<nav aria-label>`, `aria-current="page"`, 64px touch targets
- **Auth screen** — proper `<Label htmlFor>` associations, `<form>` element, show/hide password has `aria-label`
- **WeeklyCheckIn labels** — 20+ inputs all properly labeled with `htmlFor`
- **CheckInModal a11y** — `role="dialog"`, `aria-modal`, custom checkbox with `role="checkbox"`, `aria-checked`
- **Focus-visible styles** — keyboard focus indicators defined for all interactive elements
- **Toast accessibility** — `role="status"`, `aria-live="polite"`
- **Sentry masking** — sensitive elements use `data-sentry-mask`
- **External links** — `rel="noopener noreferrer"` used consistently
- **PWA setup** — well-configured service worker, reasonable precache size (1.36MB / 75 files)
- **Image optimization** — all public images are small and appropriate
- **Code splitting** — Coach.tsx lazy-loaded, screens split into separate chunks
- **Store tests** — macroStore (18), xpStore (20), workoutStore (22) are thoroughly tested
- **E2E infrastructure** — Playwright with localStorage seeding, animation disabling, proper timeouts
